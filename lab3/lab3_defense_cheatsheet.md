# Шпаргалка для защиты лабораторной №3

## Рассказ на 1 минуту

Я взял учебный проект `vuln_django_play`, который используется в моём DevSecOps-репозитории. Сначала собрал инвентаризацию компонентов по `requirements.txt`, `Dockerfile`, `docker-compose.yml`, `docker-micro.yml`, nginx-конфигам и Django settings. Потом для ключевых компонентов проверил CVE в NVD, GitHub Security Advisories и OSV. В реестр добавлял только те CVE, где наша версия попадает в уязвимый диапазон. Для каждой находки отдельно указал применимость: где компонент используется, можно ли атаковать через текущий стенд и что исправить. Результаты подготовлены для ручного ввода или импорта в DefectDojo.

## Что сказать про DefectDojo

DefectDojo использовался как система учёта результатов. В нём создаётся структура:

```text
Product -> Engagement -> Test -> Findings
```

Находки можно внести вручную через `Add Finding` или импортировать JSON через:

```text
Engagement -> Import Scan Results -> Generic Findings Import
```

В DefectDojo важно заполнить не только CVE, но и описание применимости, mitigation, severity, component name и component version.

## Топ-10 вопросов и ответы

### 1. Чем отличаются CVE, CWE, CVSS, CPE?

CVE — конкретная уязвимость. CWE — класс ошибки. CVSS — численная оценка серьёзности. CPE — стандартизированное имя продукта и версии, которое помогает NVD сопоставлять CVE с компонентами.

### 2. Почему CVSS сам по себе плохой критерий приоритизации?

CVSS не знает контекст моего стенда. Например, CVE может быть High, но уязвимая функция не используется. Или наоборот, Medium может быть важнее, если компонент смотрит наружу и реально доступен атакующему.

### 3. Что такое EPSS?

EPSS — прогноз вероятности эксплуатации CVE. Он помогает понять, насколько вероятно, что уязвимость будут использовать в реальных атаках.

### 4. Как проверить применимость CVE?

Нужно проверить версию, уязвимый диапазон, где компонент используется, доступна ли уязвимая функция извне, есть ли нужная конфигурация и можно ли построить сценарий атаки на текущем стенде.

### 5. Чем отличаются NVD, БДУ ФСТЭК и GHSA?

NVD — база NIST с CVE, CVSS, CPE и CWE. БДУ ФСТЭК — российская база уязвимостей и угроз. GHSA — GitHub Security Advisories, удобна для open source пакетов npm, PyPI, Maven и других экосистем.

### 6. Что такое транзитивная зависимость?

Это зависимость, которую проект не использует напрямую, но она подтягивается другой библиотекой. Например, `urllib3` используется библиотекой `requests`.

### 7. Чем отличается уязвимость приложения от уязвимости базового образа?

Уязвимость приложения находится в коде или библиотеке проекта. Уязвимость базового образа находится в ОС или пакетах контейнера. Например, проблема в Django — это уровень приложения, а проблема в Debian-пакете внутри `python:3.9-bookworm` — уровень образа.

### 8. Что делать, если версия уязвима, но функция не используется?

Нужно честно указать статус `Неприменимо` или `Требует дополнительной проверки`, объяснить почему и оставить рекомендацию. Полностью игнорировать не стоит, потому что конфигурация может измениться.

### 9. Что такое компенсирующая мера?

Это мера, которая снижает риск без немедленного обновления компонента. Например, закрыть `/admin/` по IP allowlist, настроить nginx на отклонение неоднозначных TE/CL-запросов или отключить опасную функцию.

### 10. Зачем нужен DefectDojo?

DefectDojo нужен, чтобы вести находки как рабочий процесс: хранить описание, статус, критичность, рекомендации, историю исправления и результаты повторной проверки. Это удобнее, чем держать CVE только в Markdown-таблице.

## Коротко про топ-3 находки

1. `CVE-2024-1135` в Gunicorn — важна, потому что Gunicorn реально обслуживает Django за nginx.
2. `CVE-2024-6827` в Gunicorn — похожий риск request smuggling, нужно проверить через внешний порт `8020`.
3. `CVE-2023-30608` в sqlparse — важна из-за старой версии и наличия `/polls/sql/`, хотя прямой путь через sqlparse нужно подтвердить.

## Что обязательно не забыть

- Для `nginx:latest`, `postgres:latest` и `python:3.9-bookworm` точные digest не были подтверждены, потому что Docker daemon не был запущен.
- По ним перед защитой нужно выполнить `docker images` и `docker image inspect`.
- Не говорить, что все CVE применимы. В отчёте часть CVE специально помечена как неприменимая.
