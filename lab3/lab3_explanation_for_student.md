# Объяснение для защиты лабораторной №3

## Что такое управление уязвимостями

Управление уязвимостями — это не разовая проверка и не просто список CVE. Это процесс: сначала нужно понять, какие компоненты есть в проекте, потом найти известные уязвимости для этих версий, проверить применимость к своему стенду, расставить приоритеты и завести результаты в систему учёта, например DefectDojo.

В этой лабораторной я делал именно такой процесс: взял проект `vuln_django_play`, посмотрел `requirements.txt`, `Dockerfile`, compose-файлы, nginx-конфиги и Django settings, после этого составил реестр компонентов и реестр уязвимостей.

## Чем эта лабораторная отличается от эксплуатации web-уязвимостей

Во второй лабораторной основная задача была доказать конкретную web-уязвимость: SQL injection, XSS, SSRF, path traversal и так далее.

В третьей лабораторной фокус другой. Здесь я смотрю не столько ошибки в коде приложения, сколько используемые компоненты: фреймворк Django, Gunicorn, sqlparse, requests, urllib3, nginx, PostgreSQL, базовый Docker-образ. Даже если код приложения сам не содержит новой ошибки, старая версия библиотеки может быть уязвимой.

## Что такое компонент

Компонент — это часть приложения или окружения, от которой зависит работа стенда. Это может быть:

- библиотека из `requirements.txt`;
- framework, например Django;
- runtime, например Python;
- web-сервер, например nginx;
- база данных, например PostgreSQL или SQLite;
- Docker-образ;
- middleware или драйвер БД.

Компонент важно фиксировать с версией. Без версии нельзя корректно понять, относится CVE к стенду или нет.

## Что такое CVE

CVE — это идентификатор конкретной публично известной уязвимости. Например, `CVE-2024-1135` — это не класс ошибки вообще, а конкретная уязвимость в Gunicorn.

CVE нужен, чтобы разные базы и инструменты говорили об одной и той же проблеме одинаковым номером.

## Что такое CVSS

CVSS — это численная оценка опасности уязвимости от 0 до 10. Она описывает общие технические свойства: можно ли атаковать по сети, нужны ли права, нужна ли реакция пользователя, какой ущерб для конфиденциальности, целостности и доступности.

CVSS полезен, но его недостаточно. Например, CVE может иметь High, но быть неприменимой к стенду, если нужная функция не используется.

## Что такое CWE

CWE — это класс слабости. Например:

- `CWE-79` — XSS;
- `CWE-89` — SQL Injection;
- `CWE-22` — Path Traversal;
- `CWE-444` — HTTP Request Smuggling.

CVE отвечает на вопрос “какая конкретная уязвимость”, а CWE — “какой тип ошибки”.

## Что такое CPE

CPE — это стандартизированное имя продукта и версии. NVD использует CPE, чтобы машинно сопоставлять CVE с продуктами.

Пример смысла CPE: “Django от djangoproject такой-то версии”. Вручную я обычно не пишу CPE в отчёт, но понимаю, что NVD использует его для привязки уязвимости к версии.

## Что такое EPSS

EPSS — это оценка вероятности эксплуатации CVE в ближайшее время. CVSS показывает техническую опасность, а EPSS пытается оценить вероятность, что уязвимость реально будут эксплуатировать.

Для приоритизации EPSS полезен: если CVSS высокий и EPSS высокий, исправлять надо быстрее.

## Что такое CISA KEV

CISA KEV — это каталог уязвимостей, которые уже подтверждённо эксплуатируются в реальных атаках. Если CVE есть в KEV, это сильный сигнал: такую уязвимость нельзя откладывать только потому, что “у нас вроде маленький стенд”.

Для найденных CVE я отдельно учитывал, надо ли проверить KEV. В отчёте оставлены источники для ручной сверки.

## Что такое DefectDojo

DefectDojo — это система для ведения находок безопасности. В ней можно создать:

```text
Product -> Engagement -> Test -> Finding
```

В лабораторной DefectDojo нужен, чтобы не хранить CVE просто в таблице, а вести их как находки: с критичностью, статусом, описанием, рекомендацией и жизненным циклом.

## Зачем нужна проверка применимости

Проверка применимости нужна, потому что CVE может относиться к версии компонента, но не быть достижимой на конкретном стенде.

Пример из этой работы: Django 3.0.4 попадает под CVE, связанную с Memcached. Но в `settings.py` Memcached не настроен, а в compose-файлах нет Memcached-сервиса. Значит версия уязвима по диапазону, но конкретный сценарий на стенде не подтверждается.

Другой пример: Gunicorn 20.0.4 уязвим к request smuggling. Он реально используется как WSGI-сервер за nginx. Поэтому эта находка важнее, чем CVE в библиотеке `requests`, которая установлена, но не импортируется кодом приложения.

## Почему нельзя просто копировать все CVE подряд

Если скопировать все CVE по слову “Django”, отчёт будет выглядеть большим, но бесполезным. Преподаватель может спросить: “Где это используется? Какая версия? Как атаковать именно ваш стенд?” Если ответа нет, находка слабая.

Правильный подход:

1. Найти компонент и версию в файлах проекта.
2. Найти CVE в NVD, OSV, GHSA или другом источнике.
3. Проверить уязвимый диапазон.
4. Проверить, используется ли функция на стенде.
5. Записать сценарий эксплуатации или честно поставить “неприменимо” / “требует проверки”.

Так отчёт получается меньше похож на список из сканера и больше похож на реальную работу по управлению уязвимостями.
