# Лабораторная №2 — короткие ответы для устной защиты

## 1. Почему работа считается выполненной, если File Upload не дал RCE?

Работа считается выполненной, потому что 7 стендов подтверждены живыми ответами сервера, а по шкале этого достаточно для максимального балла.  
По File Upload я показал обход blacklist через `.php3`.  
Финальный RCE не случился не из-за защиты, а из-за ошибки окружения: отсутствовала директория `uploads/`.  
**Пример:** сервер вернул `move_uploaded_file(... uploads/upload-shell.php3): No such file or directory`.

## 2. Чем вариант А отличается от варианта Б?

Вариант А — это практическая эксплуатация: я запускаю payload и показываю ответ сервера.  
Вариант Б — это анализ без рабочего exploit: объяснить код, payload в теории и fix.  
У меня в основном вариант А, потому что 7 стендов подтверждены командами.  
**Пример:** для Path Traversal команда с `/etc/passwd` реально вернула содержимое файла.

## 3. Почему CSRF не равен XSS?

CSRF заставляет браузер жертвы отправить запрос с её cookie.  
XSS внедряет JavaScript или HTML в страницу.  
При CSRF код в браузере жертвы выполнять не обязательно.  
**Пример:** стенд `1331` меняет email через POST с cookie `session=123456`.

## 4. Почему CORS не защищает от CSRF?

CORS ограничивает чтение ответа JavaScript-кодом с другого домена.  
Но сам запрос браузер всё равно может отправить.  
Для CSRF опасно именно выполнение действия, а не чтение ответа.  
**Пример:** HTML-форма может отправить POST на `1331`, даже если ответ нельзя прочитать из JS.

## 5. Почему Base64 не является защитой?

Base64 — это просто кодировка, а не шифрование и не подпись.  
Любой может декодировать строку, изменить JSON и закодировать обратно.  
Сервер не проверяет, кто создал эти данные.  
**Пример:** cookie `userdata` с Base64 от `{"role":"admin"}` открыла `Admin dashboard`.

## 6. В чём разница между authentication и authorization?

Authentication отвечает на вопрос: “кто ты?”.  
Authorization отвечает: “что тебе разрешено?”.  
В лабораторной чаще ломалась именно authorization.  
**Пример:** на стенде `1333` сервер поверил роли `admin` из cookie без нормальной проверки прав.

## 7. Что такое IDOR?

IDOR — это доступ к объекту или роли через прямую подмену параметра, id или cookie.  
Проблема в том, что сервер доверяет данным от клиента.  
Права надо проверять на сервере для каждого объекта.  
**Пример:** `user=tom` в cookie позволил читать `details/1337.json` и `details/4242.json`.

## 8. Почему ошибка `&&` опасна?

Потому что условие становится слишком мягким.  
Доступ запрещается только если сразу две проверки плохие.  
Если одна часть выглядит правильной, доступ открывается.  
**Пример:** на стенде `1334` достаточно поставить `user=tom`, даже с неправильной сессией.

## 9. Почему `realpath()` лучше удаления `../`?

Удаление `../` работает только против одного вида payload.  
А `realpath()` показывает настоящий путь файла после нормализации.  
После этого можно проверить, что файл внутри разрешённой папки.  
**Пример:** `/etc/passwd` не содержит `../`, поэтому фильтр стенда `1335` его пропустил.

## 10. Почему prepared statements лучше `real_escape_string`?

Prepared statements разделяют SQL-код и данные.  
Данные уже не могут стать частью SQL-команды.  
`real_escape_string` легко применить не туда или забыть.  
**Пример:** на стенде `1336` экранированная копия была создана, но в SQL попал исходный `$color`.

## 11. Что такое blind SQL Injection?

Это SQL-инъекция, где база не выводит данные напрямую.  
Мы задаём вопросы true/false и смотрим, меняется ли ответ приложения.  
Так можно постепенно восстановить данные.  
**Пример:** `OR 1=1` дал `10 exist`, а `OR 1=2` дал `OUT OF STOCK`.

## 12. Что такое SSRF?

SSRF — это когда пользователь заставляет сервер сделать запрос вместо себя.  
Опасность в доступе к внутренним адресам, недоступным снаружи.  
Сервер становится “прокси” для атакующего.  
**Пример:** на стенде `1337` URL `http://127.1:1337/` заставил backend сходить на loopback.

## 13. Почему allowlist лучше blacklist?

Blacklist пытается перечислить всё плохое, но легко что-то забыть.  
Allowlist разрешает только заранее известные безопасные варианты.  
Это проще контролировать.  
**Пример:** upload blacklist забыл `.php3`, а SSRF blacklist забыл `127.1`.

## 14. Чем `Array.includes` отличается от `String.includes`?

`String.includes("<")` ищет символ внутри строки.  
`Array.includes("<")` ищет отдельный элемент массива, равный `<`.  
Из-за этого массив с HTML внутри проходит проверку.  
**Пример:** `letterContent[]=<img ...>` обошёл фильтр на стенде `1338`.

## 15. Как CSP снижает вред от XSS?

CSP ограничивает, какие скрипты можно выполнять на странице.  
Даже если HTML-инъекция прошла, CSP может заблокировать inline-script или загрузку чужого JS.  
Но CSP не заменяет исправление XSS.  
**Пример:** на стенде `1338` CSP могла бы помешать выполнению `onerror=alert(...)`.

## 16. Какой общий вывод по лабораторной?

Главная проблема во всех стендах — доверие к пользовательским данным.  
Сервер должен проверять типы, права, пути, SQL-параметры и URL на своей стороне.  
Исправление должно закрывать класс уязвимости, а не один payload.  
**Пример:** нужны CSRF-токены, server-side authorization, `realpath()`, prepared statements, allowlist и HTML escaping.
